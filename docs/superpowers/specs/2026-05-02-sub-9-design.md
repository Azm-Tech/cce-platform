# Sub-Project 09 — Smart Assistant streaming + threading + citations — Design Spec

**Date:** 2026-05-02
**Status:** Approved
**Predecessor:** [Sub-8 Interactive City](2026-05-02-sub-8-design.md)
**Target release:** `web-portal-v0.4.0`
**Backend changes:** modest (one endpoint changes shape; the stub assistant client is enhanced).

---

## Goal

Replace the Sub-6 Phase 9 single-turn assistant skeleton at `/assistant` with a real conversational UI: multi-turn threading, server-streamed responses (SSE), and structured citations grounded in the existing knowledge maps + resources. The LLM itself stays a **fake-streaming stub** for v0.1.0 — the architecture is designed so the real LLM client is a one-class swap-in later (Sub-10+) with zero frontend changes.

## Non-goals

- **Real LLM integration** — the backend stub fake-streams chunks of placeholder text and emits 1–2 fake citations from seeded data. This is intentional: it lets us ship the full UX, exercise streaming + citation flows end-to-end in tests, and avoid coupling Sub-9 to LLM-vendor selection / cost / prompt design / rate limiting / RAG implementation.
- **Server-persisted threads** — conversations live in memory only. When the user navigates away or refreshes, the thread is gone. Real persistence lands with the real LLM (Sub-10+) along with whatever auth/identity model real conversations need.
- **Markdown rendering of assistant replies** — plain text in v0.1.0. The stub never emits markdown anyway; revisit when the real LLM is wired up.
- **Code blocks, math rendering, image attachments** — also deferred; YAGNI for the stub.
- **Multi-thread management UI (sidebar of past chats)** — single thread per session.
- **Sharing / export of conversations** — defer.
- **Voice / TTS** — out of scope.

## Architecture

Single route `/assistant`. Mirrors Sub-7's `MapViewerPage` and Sub-8's `ScenarioBuilderPage`: the page provides an `AssistantStore` at the component level, and sub-components consume signals + emit actions. No prop drilling.

## Data contracts

### Backend wire format (changes from Sub-6 Phase 9)

```ts
// POST /api/assistant/query
interface AssistantQueryRequest {
  // Full conversation history. The most recent entry is the user message
  // we want a reply to. The server inspects messages[messages.length - 1].
  messages: { role: 'user' | 'assistant'; content: string }[];
  locale: 'ar' | 'en';
}

// Response: text/event-stream. Each event is `data: <json>\n\n`.
type SseEvent =
  | { type: 'text';     content: string }                          // append to current assistant message
  | { type: 'citation'; citation: Citation }                        // append to current assistant citations[]
  | { type: 'done'     }                                            // turn complete; close the stream
  | { type: 'error';    error: { kind: string } };                  // mid-stream failure

interface Citation {
  id: string;                    // GUID — stable id of the underlying resource / map node
  kind: 'resource' | 'map-node';
  title: string;                 // localized server-side from the request locale
  href: string;                  // '/knowledge-center/resources/<slug>' or '/knowledge-maps/<id>?node=<id>'
  sourceText?: string;           // optional excerpt the assistant grounded on (≤200 chars)
}
```

## Data flow

### Sending a message
1. User types in `ComposeBox`, hits Enter (or clicks send).
2. `AssistantStore.sendMessage(content)`:
   - Appends a `user` message with `status: 'complete'`.
   - Appends an `assistant` placeholder with `status: 'pending'`, empty content, no citations.
   - Sets `streaming: true`.
   - Builds an `AbortController` for the stream.
   - Calls `api.query({ messages: [...history], locale })` — returns `AsyncIterable<SseEvent>`.
3. The store iterates events:
   - `text` → append `content` to the placeholder; on first text chunk, flip status to `'streaming'`.
   - `citation` → push to placeholder's `citations[]`.
   - `done` → flip status to `'complete'`, set `streaming: false`.
   - `error` → flip status to `'error'`, set `errorKind`, set `streaming: false`.
4. On any thrown exception during iteration (network drop, parser error), same as `error` event.

### Cancel
- User clicks Cancel during streaming. Store calls `abortController.abort()`. The fetch reader unwinds; the in-flight assistant message keeps whatever text arrived and flips to `'complete'` (not `'error'` — the user chose to stop). `streaming` flips to `false`.

### Retry
- Available on the most recent message when `status: 'error'`. Removes the failed assistant message, then re-runs `sendMessage` with the same payload (the last user message is already in the history).

### Regenerate
- Available on the most recent successful assistant message. Removes that assistant message, then re-runs `sendMessage` (which finds the prior user message at the tail of the history). Used for "give me another answer."

### Clear thread
- Wipes `messages()` to `[]`. Confirms via `ConfirmDialogComponent` (reused from Sub-8) only if `messages().length > 0`.

### URL deep-link
- `assistant.page.ts` reads `?q=` from `ActivatedRoute.snapshot.queryParamMap` on `ngOnInit`. If `q` is non-empty AND `messages().length === 0`, calls `store.sendMessage(q)` immediately. Used for entry from search results, knowledge-map node detail, etc.

## Components

### `AssistantPage`
Layout host:
- Header strip with title (`assistant.title`) + Clear-thread button (disabled when empty).
- Message list fills the available scroll area.
- Sticky compose box at the bottom.
- Provides `AssistantStore`. Reads `?q=` on init.

### `MessageListComponent`
- `<ul>` rendering each message via `MessageBubbleComponent`.
- `aria-live="polite"` so screen readers hear streaming tokens.
- Auto-scroll to bottom on new message via `effect()` watching `store.messages()` length.
- Renders `TypingIndicatorComponent` after the last message when its status is `'pending'` (request open, no first chunk yet).
- Empty state: localized hint text.

### `MessageBubbleComponent`
- Single message as a `<li>` with role-keyed CSS class.
- Content area renders the text. While streaming, a blinking `▌` cursor at the tail.
- Citation footer: list of `CitationChipComponent` chips when `citations.length > 0`.
- Hover actions: Copy (clipboard), Regenerate (only on the last successful assistant message), Retry (only on a failed assistant message).
- `aria-busy="true"` while `status` is `pending`/`streaming`.

### `ComposeBoxComponent`
- Reactive `FormControl<string>` bound to a `<textarea>`.
- Auto-resizing height (max ~6 lines) via a CSS technique (`field-sizing: content` where supported, JS fallback).
- `Enter` sends; `Shift+Enter` inserts newline.
- Send button: disabled when input empty or `!store.canSend()`.
- During streaming: send button morphs into a Cancel button (calls `store.cancel()`).
- Char counter visible at ≥1500; submit disabled at >2000.

### `CitationChipComponent`
- Inline variant: `[N]` numeric marker.
- Footer variant: `[N] Title` chip with a leading kind icon (`description` for resource, `account_tree` for map-node).
- `<button>` element with `routerLink` (Angular's `RouterLink` directive on a button works via `[routerLink]`); navigates on click + Enter.
- Tooltip via `matTooltip` showing the title + optional `sourceText`.

### `TypingIndicatorComponent`
- Three dots, staggered keyframe animation. Pure CSS.
- Used in `MessageListComponent` after the last message when status is `'pending'`.

## URL state

`?q=<urlencoded question>` for deep-link entry. Removed from the URL after the message is sent (replaceUrl). No other URL state — the in-memory thread isn't reflected in the URL because it'd grow unbounded.

## Error handling

Every error surface routes through `Result<T> + toFeatureError`:
- **Network down before first token** → assistant message status `'error'`, `errorKind: 'network'`. Retry button appears.
- **Mid-stream disconnect / fetch aborts unexpectedly** → partial content stays visible; status `'error'`. Retry button appears.
- **Server emits `error` event** → same as above; `errorKind` carries the server-supplied kind.
- **User cancels** → status `'complete'` (deliberate stop, not failure). No retry button.
- **Send while still streaming** → guarded by `canSend()`; the UI never enables it. The action no-ops if called programmatically.
- **Browser offline** → `sendMessage` immediately produces an `'error'` message via the `network` `kind` from `toFeatureError`. The store doesn't open the stream.

## Accessibility

- `aria-live="polite"` on the message list — screen readers hear streaming tokens.
- `aria-busy="true"` on the assistant message bubble while `status` is `pending`/`streaming`.
- Focus management: after the stream finishes, focus returns to the compose textarea so Enter sends the next question.
- Compose textarea has a visible `<label>`. Send/Cancel buttons have `aria-label`s for both states.
- Citation chips are `<button>`s with `routerLink` so screen readers announce them as buttons + activate on Enter.
- Streaming cursor uses `currentColor`; visible in dark mode without extra logic.
- Phase 03 axe-core gate: zero critical/serious findings on `/assistant`.

## Backend-side details

### Endpoint shape change
- Request body: `{ messages: [{ role, content }], locale }` (was `{ question, locale }`).
- Response: `Content-Type: text/event-stream; charset=utf-8`. Server writes events as `data: {json}\n\n`.
- `AllowAnonymous` stays — anyone can chat.
- Cancellation token wired through so client aborts propagate.

### `ISmartAssistantClient` interface change
```cs
public interface ISmartAssistantClient
{
    IAsyncEnumerable<SseEvent> StreamAsync(
        IReadOnlyList<ChatMessage> messages,
        string locale,
        CancellationToken ct);
}
public sealed record ChatMessage(string Role, string Content);
public abstract record SseEvent
{
    public sealed record Text(string Content)         : SseEvent;
    public sealed record Citation(CitationDto Value)  : SseEvent;
    public sealed record Done()                       : SseEvent;
    public sealed record Error(string Kind)           : SseEvent;
}
```

### Stub fake-streaming behaviour
- Build a placeholder reply string that incorporates the user's most recent message.
- Yield ~8 `Text` events with 100–200ms delays between them (`Task.Delay` honouring `ct`).
- After the first half of the chunks, yield 1 `Citation` event sourced from a randomly-chosen seeded `Resource` row (when one exists for the active locale) and 1 sourced from a `KnowledgeMapNode`.
- Yield `Done`.

This is the only place the stub-vs-real-LLM line lives — swapping in an Anthropic / OpenAI / vendor client later is one class change.

## Phasing (4 phases, ~25 tasks)

- **Phase 00 — Cross-cutting**: extend `assistant.types.ts`; build `lib/sse-client.ts` + tests; backend route signature accepts `messages[]`; backend `ISmartAssistantClient` reshaped to async-stream + stub fake-streams chunks + emits citations from seeded data; new `assistant.*` i18n keys (EN/AR); component file structure (5 sub-components + page wiring stubs).
- **Phase 01 — Backend SSE**: change `/api/assistant/query` to write SSE response; integration test against the live endpoint via `Microsoft.AspNetCore.Mvc.Testing`; stub yields ~8 chunks over ~1.2s with 1–2 citations.
- **Phase 02 — Frontend store + UI**: `AssistantStore` (signals + 5 actions + computed); `MessageListComponent` + `MessageBubbleComponent` + `ComposeBoxComponent` + `CitationChipComponent` + `TypingIndicatorComponent`; full visual styling.
- **Phase 03 — Polish + close-out**: `?q=` deep-link auto-send; clear-thread confirm dialog (reuses Sub-8's `ConfirmDialogComponent`); ADR-0049 (SSE + structured citations), ADR-0050 (client-owned in-memory thread state); completion doc; CHANGELOG entry under `web-portal-v0.4.0`; tag `web-portal-v0.4.0`.

## ADRs to land in Phase 03

- **ADR-0049 — SSE + structured citation events**: chosen over fetch-and-render because users see tokens as they arrive (lower perceived latency); citations are typed events the UI can render distinctly rather than scraping inline `[1]` markers from text.
- **ADR-0050 — Client-owned in-memory thread state**: chosen over server-persisted threads for v0.1.0 because the LLM is still a stub; persistence design lands when real conversations need an identity/auth model anyway.

## Success criteria

- An anonymous user lands on `/assistant`, types a question, sees the reply stream in token-by-token, with 1–2 citation chips at the bottom of the assistant message that link into the knowledge-center / knowledge-maps pages.
- A user lands on `/assistant?q=What+is+CCE` and the question auto-sends; reply streams in.
- Cancel during streaming halts mid-token, retains the partial reply, returns focus to the compose box.
- Retry on a failed message re-sends and replaces the failed reply.
- Regenerate on a successful message rolls back and re-streams a fresh reply.
- All flows work in both EN and AR with proper RTL bubble alignment.
- Backend stub fake-streams correctly: integration test asserts at least 5 `text` events + 1 `citation` event + 1 `done` event in order.
- Web-portal v0.4.0 ships with ~45 new tests, no regressions in the existing 445.
- No new heavy frontend dependencies.
- Real LLM swap-in later is a single-class change with zero frontend or contract changes.
