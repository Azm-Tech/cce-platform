# Sub-Project 08 — Interactive City scenario builder — Design Spec

**Date:** 2026-05-02
**Status:** Approved
**Predecessor:** [Sub-7 Knowledge Maps Design](2026-05-01-sub-7-design.md)
**Target release:** `web-portal-v0.3.0`
**No backend changes** — endpoints already shipped in Sub-4 (`external-api-v0.1.0`).

---

## Goal

Replace the Sub-6 Phase 9 skeleton at `/interactive-city` with a single-page scenario-builder workbench. The user picks technologies from a catalog, sees totals recompute live as they toggle, runs the scenario against the server for an authoritative result + localized summary, and (when authenticated) saves named scenarios for later reload. Read-only browsing for anonymous users; CRUD for authenticated.

## Non-goals

- **Side-by-side scenario comparison** — out of scope v0.1.0; saved scenarios live as separate URLs the user can flip between. Tabs (Sub-7 style) are also out of scope; the builder owns one scenario at a time.
- **Drag-and-drop catalog → cart** — click-to-toggle is enough.
- **Per-row reordering** — order does not affect totals.
- **Bilingual name split** — one name input writes to both `nameAr` and `nameEn` for v0.1.0; admin-cms can split later if real users complain.
- **Visualization (chart of carbon impact over time)** — backend exposes only point-in-time totals today; revisit if data grows.
- **Editing a saved scenario in place** — load → edit → save creates a new scenario row in v0.1.0 (server doesn't expose PATCH today, only POST/GET/DELETE on `/api/me/interactive-city/scenarios`).

## Architecture

Mirrors Sub-7's `MapViewerPage` + `MapViewerStore` shape: a single page component owns the layout and provides a signals-first store at the component level (fresh state per route activation). Sub-components consume signals and emit actions; no parent prop drilling.

## Data contracts

### Already-shipped backend types (do not modify)

```ts
// GET /api/interactive-city/technologies → CityTechnology[]
interface CityTechnology {
  id: string;                     // GUID
  nameAr: string;
  nameEn: string;
  descriptionAr: string;
  descriptionEn: string;
  categoryAr: string;
  categoryEn: string;
  carbonImpactKgPerYear: number;  // negative = reduces emissions
  costUsd: number;
  iconUrl: string | null;
}

// POST /api/interactive-city/scenarios/run (anonymous OK)
interface RunRequest {
  cityType: 'Coastal' | 'Industrial' | 'Mixed' | 'Residential';
  targetYear: number;
  configurationJson: string;      // JSON string with shape {"technologyIds": ["guid1", ...]}
}
interface RunResult {
  totalCarbonImpactKgPerYear: number;
  totalCostUsd: number;
  summaryAr: string;
  summaryEn: string;
}

// /api/me/interactive-city/scenarios (auth required)
interface SaveRequest {
  nameAr: string;
  nameEn: string;
  cityType: 'Coastal' | 'Industrial' | 'Mixed' | 'Residential';
  targetYear: number;
  configurationJson: string;
}
interface SavedScenario {
  id: string;
  nameAr: string;
  nameEn: string;
  cityType: 'Coastal' | 'Industrial' | 'Mixed' | 'Residential';
  targetYear: number;
  configurationJson: string;
  createdOn: string;              // ISO 8601
}
```

`configurationJson` is opaque to the server beyond the `technologyIds` field — future v0.x can add knobs (intensity, year-by-year ramps) without a schema migration.

### Frontend store state

```ts
class ScenarioBuilderStore {
  // Catalog (loaded once on entry, cached for the route lifetime)
  readonly technologies = signal<CityTechnology[]>([]);
  readonly catalogLoading = signal<boolean>(false);
  readonly catalogError = signal<string | null>(null);

  // Editable scenario
  readonly cityType = signal<CityType>('Mixed');           // default 'Mixed'
  readonly targetYear = signal<number>(2030);              // clamped [currentYear, currentYear+50]
  readonly name = signal<string>('');                      // single language for v0.1.0; written to both nameAr/En
  readonly selectedIds = signal<ReadonlySet<string>>(new Set());

  // Server result (populated by Run, cleared on any edit)
  readonly serverResult = signal<RunResult | null>(null);

  // Saved scenarios sidebar (auth-only)
  readonly savedScenarios = signal<SavedScenario[]>([]);
  readonly savedLoading = signal<boolean>(false);
  readonly savedError = signal<string | null>(null);

  // Network state
  readonly running = signal<boolean>(false);
  readonly saving = signal<boolean>(false);
}
```

(The triple `signal<T>` + `loading` + `error` pattern matches `KnowledgeMapsListPage` and `MapViewerStore` from Sub-6/Sub-7 — no new `RemoteData` abstraction.)

### Computed signals

- `liveTotals` — sum of `carbonImpactKgPerYear` and `costUsd` over `selectedTechnologies`. Recomputes synchronously on toggle. Source of truth between Runs.
- `selectedTechnologies` — joins `selectedIds` against `technologies` for display.
- `dirty` — true when current selection differs from the last loaded saved scenario; drives the unsaved-changes warning before `loadFromSaved` overwrites state.
- `canRun` — `selectedIds.size > 0`.
- `canSave` — `selectedIds.size > 0 && name.trim() !== ''`.

## Data flow

### On route entry
1. Store `init()` fires `GET /api/interactive-city/technologies` → catalog cached.
2. If `isAuthenticated()` (read from existing `AuthState`) → `GET /api/me/interactive-city/scenarios` → drawer hydrates. Otherwise drawer renders the sign-in CTA.
3. URL `?city=&year=&t=&name=` is parsed; store hydrates. Empty URL → defaults.

### While editing
- Toggling a catalog card calls `store.toggle(id)` → `selectedIds` mutates → `liveTotals` recomputes → `serverResult` clears (totals bar reverts to client-only view).
- Header inputs update store via reactive forms two-way binding.
- A 200ms-debounced effect writes the store back to the URL (`Router.navigate` with `queryParamsHandling:'merge'`, `replaceUrl:true`).

### Run
- `POST /api/interactive-city/scenarios/run` with request `{ cityType, targetYear, configurationJson: JSON.stringify({ technologyIds: [...selectedIds] }) }`.
- On success: `serverResult` is set; totals bar shows the server summary text underneath the numbers.
- On error: `running:false`, surface a `FeatureError`-shaped toast; `liveTotals` remains visible.

### Save
- If `!isAuthenticated()` → invoke existing `signIn()` helper (BFF cookie + Keycloak redirect; resumes on `/interactive-city?...`).
- Otherwise `POST /api/me/interactive-city/scenarios` with `{ nameAr: name, nameEn: name, cityType, targetYear, configurationJson }` → on `201`, prepend the returned scenario to `savedScenarios` and toast "Saved".

### Load saved
- `loadFromSaved(scenario)` parses `configurationJson`, sets `selectedIds`/`cityType`/`targetYear`/`name`, clears `serverResult`. Does **not** auto-run. URL re-syncs.
- If `dirty` is true, an `unsavedChanges` confirmation dialog gates the load.

### Delete saved
- Confirm dialog → `DELETE /api/me/interactive-city/scenarios/{id}` → drop from `savedScenarios`. Doesn't touch the form even if that scenario is currently loaded — the user can keep editing what they have.

## Components

### `ScenarioHeaderComponent`
A horizontal three-input strip:
- `mat-form-field` + `mat-input` for **name** (one input → both `nameAr`/`nameEn`).
- `mat-form-field` + `mat-select` for **city type** (4 options, labels via `interactiveCity.cityType.<value>`).
- `mat-form-field` + `mat-input type="number"` for **target year** (`min` = `currentYear`, `max` = `currentYear+50`, `step=1`, default 2030).

Reactive `FormGroup`. Validation: name optional during editing (required at Save time, surfaced via the Save button's enabled state); year clamped on blur; city-type required (defaults to `Mixed`). Form `valueChanges` → store actions.

### `TechnologyCatalogComponent`
Left rail:
- Search box (200ms debounced; filters by `nameEn|nameAr|categoryEn|categoryAr` — same predicate shape as Sub-7's `nodeMatches`).
- Grouped list: `mat-card` per technology under collapsible category headers. Card shows name, category chip, `carbonImpactKgPerYear` (color-coded: negative=green, positive=warn), `costUsd`. Click toggles selection (visual: checkmark + raised border + `aria-pressed="true"`).
- Empty state when search yields no matches.

### `SelectedListComponent`
Right rail:
- Header: "Selected (N)" + "Clear all" button (only enabled when `N > 0`; confirms inline if N ≥ 5 to avoid mistakes).
- List of selected technologies as compact `mat-list-item`s with name + per-row carbon/cost + remove × button (`aria-label` localized).
- Empty state: "Pick technologies from the catalog to start."

### `TotalsBarComponent`
Sticky bottom bar:
- Three columns: **Total Carbon Impact** (kg/year), **Total Cost** (USD), **Server Summary** (filled after Run; empty otherwise — column shrinks).
- `aria-live="polite"` so toggle updates announce.
- Two right-aligned buttons: **Run** (`mat-flat-button`, primary; disabled when `!canRun`; spinner while `running()`); **Save** (`mat-stroked-button`; disabled when `!canSave`; opens save dialog if `name` is empty, else saves with current name; if not auth → triggers `signIn()`).

### `SavedScenariosDrawerComponent`
- Right side rail desktop / bottom sheet mobile (CSS-driven, 720px media query — same pattern as `NodeDetailPanelComponent`).
- `role="complementary"` + `aria-expanded`.
- Authenticated: list of saved scenarios with name, city-type chip, year, "Load" button, "Delete" icon. Confirm dialog on delete.
- Unauthenticated: single CTA card "Sign in to save scenarios" with a button that calls `signIn()`.

## URL state

`?city=<CityType>&year=<int>&t=<guid>,<guid>,...&name=<urlencoded>`

`lib/url-state.ts`:
- `parseUrlState(params: ParamMap): Partial<ScenarioBuilderState>` — drops unknowns, clamps year, ignores non-GUID `t` entries, defaults invalid city, decodes `name`.
- `buildUrlPatch(state: ScenarioBuilderState): Params` — round-trips on `parse(build(s)) === s` for valid states; tested.

Round-trips on hydrate ↔ store via a single effect inside the store, debounced 200ms.

## Error handling

All API calls go through `Result<T> + toFeatureError`. Error surfaces:

- **Catalog load fails** — full-page error block with retry button (the page is useless without it).
- **Saved scenarios load fails** — drawer renders inline "Couldn't load — retry"; catalog still works.
- **Run fails** — toast; `liveTotals` stays visible (the client has the math).
- **Save fails: 401** — silently kicks off `signIn()`; on resume, re-attempts the save.
- **Save fails: other** — toast "Couldn't save — try again"; form keeps state.
- **Delete fails** — toast; the row stays.
- **Browsers offline** — every interactive feature degrades gracefully because `liveTotals` doesn't need the server.

## Accessibility

- Catalog and selected list are real `<ul>` lists with `<button>` rows (focusable; `aria-pressed` on selected catalog cards; `aria-label` on remove buttons).
- Reactive Form provides `<label>` association out of the box.
- Totals bar uses `aria-live="polite"` so screen readers hear updates after toggle.
- Drawer uses `role="complementary"` + `aria-expanded`.
- Keyboard: Tab cycles header → catalog → selected list → totals → drawer; Enter/Space toggles selection in the catalog; Esc closes the drawer; Esc on focused dialog cancels.
- Phase 5 axe-core gate: zero critical/serious findings on `/interactive-city` (auth and anon).

## i18n + RTL

New keys under `interactiveCity.*`:

- `interactiveCity.builder.{title,subtitle,name,cityType,targetYear,unsavedChangesTitle,unsavedChangesBody,unsavedChangesConfirm,unsavedChangesCancel}`
- `interactiveCity.cityType.{Coastal,Industrial,Mixed,Residential}`
- `interactiveCity.catalog.{title,search,empty,addLabel,categoryAll}`
- `interactiveCity.selected.{title,empty,clear,clearConfirm,remove}`
- `interactiveCity.totals.{carbon,cost,run,running,save,saving,summaryEmpty}`
- `interactiveCity.saved.{title,empty,load,delete,confirmDelete,confirmDeleteBody,signInToSave,signIn}`
- `interactiveCity.errors.{loadCatalog,loadSaved,runFailed,saveFailed,deleteFailed,retry}`

Both EN and AR. RTL is automatic via `LocaleService` and Material's RTL handling — no per-component mirroring logic (the canvas/coordinate problem from Sub-7 doesn't apply here).

## Phasing (5 phases, ~30 tasks)

- **Phase 00 — Cross-cutting**: extend `interactive-city.types.ts` (RunResult, SavedScenario, RunRequest, SaveRequest); extend `InteractiveCityApiService` with `runScenario / listMyScenarios / saveScenario / deleteMyScenario`; create `lib/url-state.ts` + tests; new i18n keys (EN/AR); declare component file structure; Material module imports.
- **Phase 01 — Store + page shell**: `ScenarioBuilderStore` with init + computed live totals + URL hydrate effect; `ScenarioBuilderPage` skeleton with header + empty layout slots; replaces Sub-6 Phase 9 page; route is unchanged.
- **Phase 02 — Catalog + selection**: `TechnologyCatalogComponent` with search + grouping + click-to-toggle; `SelectedListComponent` with remove + clear; visible selection state; live totals updating in a placeholder bar.
- **Phase 03 — Run + save**: `TotalsBarComponent` with Run + Save buttons; save dialog; `runScenario` integration including server summary text; auth-gated save with sign-in fallback.
- **Phase 04 — Saved scenarios drawer**: `SavedScenariosDrawerComponent` (auth-only) with load/delete; CSS drawer (right rail desktop / bottom sheet mobile via 720px media query); load-with-unsaved-changes confirmation.
- **Phase 05 — Polish + ADRs + completion**: a11y audit (axe-core in tests); URL deep-link smoke check; ADR-0047 (single-page workbench rationale); ADR-0048 (client-side live totals + server-authoritative on Run); `web-portal-v0.3.0` completion doc + tag; CHANGELOG entry.

## ADRs to land in Phase 5

- **ADR-0047 — Single-page workbench (no wizard)**: chosen over multi-step wizard because the tool is exploratory; live recomputation matters more than guided structure.
- **ADR-0048 — Client-side live totals, server-authoritative on Run**: avoids a network call per toggle while still validating math against server state at commit time.

## Success criteria

- An anonymous user can land on `/interactive-city`, pick technologies, see live totals update, and click Run to get a localized summary string from the server.
- A user lands on `/interactive-city?city=Industrial&year=2035&t=guid,guid&name=My%20City` and the page hydrates exactly that scenario.
- An authenticated user can save a named scenario, see it appear in the drawer, navigate away, return, and reload it.
- All flows work in EN and AR (with proper RTL layout).
- Lighthouse a11y score on `/interactive-city` ≥ 90.
- Web-portal v0.3.0 ships with ~50 new tests, no regressions in Sub-7's 362 web-portal tests.
- No backend changes; no new heavy dependencies.
