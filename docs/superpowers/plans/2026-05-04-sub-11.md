# CCE Sub-Project 11 — Entra ID migration — Implementation Plan (Master)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan phase-by-phase. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace Keycloak with Microsoft Entra ID across the entire CCE platform. Multi-tenant configuration accepts CCE-staff users (synced from `cce.local` AD via Entra ID Connect) + external partner users (signing in from their own Entra ID tenants). CCE backend creates users via Microsoft Graph for self-service registration. Sign-in pages branded for the CCE tenant. Conditional Access enforces MFA. Big-bang cutover swaps all 4 envs in a single maintenance window.

**Architecture:** `Microsoft.Identity.Web` 3.5.0 layers on top of the existing `Microsoft.AspNetCore.Authentication.JwtBearer` + `OpenIdConnect` 8.0.10 packages — both APIs and both BFFs (web-portal, admin-cms) consume Entra ID's OIDC discovery + JWKS. A custom `IssuerValidator` accepts any tenant matching `https://login.microsoftonline.com/<any-tenant>/v2.0`. `EntraIdRegistrationService` calls `Microsoft.Graph` 5.65.0 to create users in CCE's own tenant (option b — `User.ReadWrite.All` application permission). New nullable `EntraIdObjectId` column on `Users` table; `EntraIdUserResolver` lazy-links UPN → objectId on first sign-in. App roles drive permissions via the unchanged `permissions.yaml` matrix. Conditional Access enforces MFA at the Entra ID side; app stays MFA-agnostic. Tests use `WireMock.Net` 1.7.0 with recorded JSON fixtures (PII-scrubbed); no real Entra ID at test time.

**Tech Stack:** .NET 8 minimal APIs (existing); `Microsoft.Identity.Web` 3.5.0 + `Microsoft.Identity.Web.MicrosoftGraph` 3.5.0 + `Microsoft.Identity.Web.UI` 3.5.0 (NEW); `Microsoft.Graph` 5.65.0 (NEW); `Microsoft.AspNetCore.Authentication.JwtBearer` + `OpenIdConnect` 8.0.10 (existing, retained); `WireMock.Net` 1.7.0 (NEW, replaces Testcontainers Keycloak); EF Core 8 migration adds nullable `EntraIdObjectId` GUID column on `Users` table.

**Spec reference:** [`../specs/2026-05-04-sub-11-design.md`](../specs/2026-05-04-sub-11-design.md) — 8 scope decisions, 5-phase plan, ~28 tasks.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Multi-tenant `IssuerValidator` rejects valid tokens (false negative) | Exhaustive unit tests in `EntraIdJwtAuthTests` cover the validator's regex shape (`https://login.microsoftonline.com/<any>/v2.0`). Recorded fixtures from real Entra ID dev tenant exercise actual issuer strings. |
| Recorded fixtures drift when Entra ID changes its API surface | Fixtures pinned to a recorded date; if Entra ID returns differently in production, integration tests at cutover surface it. Refresh procedure documented in `Identity/Fixtures/entra-id-fixtures/README.md`. |
| `EntraIdObjectId` column collision: two users get the same objectId via concurrent first-sign-in | Filtered unique index `IX_Users_EntraIdObjectId` enforces uniqueness at the DB layer. Resolver catches `DbUpdateException`, re-reads, and treats the second attempt as a no-op. |
| Self-service registration succeeds in Graph but fails in CCE.DB persist | `ProfileEndpoints` registration handler catches the persist error AFTER Graph success and surfaces a 500 to the caller. Operator runbook documents the recovery: locate the orphaned Entra ID user, decide whether to delete from Graph or persist the CCE-side row manually. Future Sub-11+ improvement: idempotent registration with retry. |
| BFF `OnTokenValidated` event fails (DB unreachable during sign-in) | Resolver wraps `_db.SaveChangesAsync()` in try/catch; logs error but does NOT block sign-in. Subsequent sign-in retries the link. Worst case: user signs in but role-checks may fail until linkage exists. |
| Microsoft Graph throttling (Graph rate limits per app per tenant) | `Microsoft.Identity.Web`'s default token cache + Graph SDK's automatic retry handle most cases. Registration endpoint rate-limited at CCE side (existing `BffOptions.AntiSpam` + `RateLimiter` middleware) prevents flood. |
| Entra ID Premium licensing gates branding + Conditional Access | Documented as prerequisite in `infra/entra/README.md`. Sub-11 doesn't ship without P1/P2; cutover blocked until licensing is in place. |
| Cutover fails partway through; some envs on Entra ID, some on Keycloak | `rollback.ps1 -ToTag <pre-Sub-11-tag>` per env reverts in seconds; forward-only-friendly migration means old image runs against new schema. Documented in `entra-id-cutover.md` rollback section. |
| External partner can't sign in because their tenant admin hasn't consented | Documented in `infra/entra/README.md` partner-tenant onboarding section. Admin-consent URL provided per partner. |
| Existing tests break because of `RoleClaimType` change (`groups` → `roles`) | Phase 00 Task 0.1 inventories existing role-claim test usage and migrates them; final Phase 00 close-out verifies all baseline tests still pass. |
| `BffTokenRefresher.cs` deletion breaks call sites | Phase 01 Task 1.4 inventories call sites BEFORE deletion; if any consumers exist, refactor them to use `ITokenAcquisition` instead. Most likely BffTokenRefresher is only called from `BffAuthEndpoints` which is being modified anyway. |

## Out of scope (deferred to Sub-11d+ or beyond)

- **AD FS** — explicitly rejected at scope-decision time (Microsoft is deprecating).
- **B2C consumer accounts** — multi-tenant covers partner orgs; B2C is wrong shape.
- **Single-tenant** — locked to multi-tenant per Decision 1.
- **Programmatic Conditional Access policy provisioning** — admin-driven via Entra ID portal.
- **Distributed token cache** (Redis-backed) — in-memory cache OK for single-host deploy.
- **Branding for partner tenants** — Entra ID security boundary; impossible.
- **Migration of existing Keycloak MFA registrations** — users re-enroll under CA on first sign-in.
- **AD FS server provisioning** — out of scope by virtue of choosing Entra ID.
- **Service-to-service auth between APIs** — backend network policies + per-API JWT bearer remain unchanged.
- **SSO with non-Microsoft IdPs** — partners federate via their own tenant's IdP integrations.
