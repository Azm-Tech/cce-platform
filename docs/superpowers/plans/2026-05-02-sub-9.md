# CCE Sub-Project 09 — Smart Assistant streaming + threading + citations — Implementation Plan (Master)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan phase-by-phase. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Sub-6 Phase 9 single-turn assistant skeleton at `/assistant` with a real conversational UI: multi-turn threading, server-streamed responses (SSE), and structured citations grounded in the existing knowledge maps + resources. The LLM stays a fake-streaming stub for v0.1.0; the architecture is designed so a real LLM client is a one-class swap-in later.

**Architecture:** All new frontend surface lives under `apps/web-portal/src/app/features/assistant/`. A single `AssistantStore` (signals-first, provided per-route) owns the in-memory thread, the `AbortController` for the in-flight stream, and the network flags. Sub-components consume signals + emit actions. Backend `POST /api/assistant/query` grows to accept `messages[]` and respond with `text/event-stream`; `ISmartAssistantClient` is reshaped to `IAsyncEnumerable<SseEvent>` and the stub fake-streams chunks + emits 1–2 citations from seeded data. No persistence — threads are gone on refresh.

**Tech Stack:** Angular 19 standalone, signals-first, Reactive Forms, Material 18, ngx-translate ar/en + RTL, native `fetch` + `ReadableStream` for SSE (no `eventsource` polyfill), Jest 30 (unit), Playwright + `@axe-core/playwright` (E2E + a11y), .NET 8 minimal APIs + MediatR + `IAsyncEnumerable<SseEvent>` for streaming.

**Spec reference:** [`../specs/2026-05-02-sub-9-design.md`](../specs/2026-05-02-sub-9-design.md) — 13 sections, 4-phase plan, ~25 tasks.

### Patterns inherited from Sub-6 / Sub-7 / Sub-8

- **Result<T> + toFeatureError** — every API call returns `{ ok: true, value } | { ok: false, error }`; helper from `@frontend/ui-kit`. Streaming methods return an `AsyncIterable<SseEvent>` directly; errors surface as `error` events the store maps to `FeatureError`.
- **Signals-first store** — three-signal-per-resource shape where applicable (`signal<T>` + `loading: signal<boolean>` + `error: signal<string|null>`); computed signals derive everything else. The Sub-9 thread state is simpler — one `messages: signal<ThreadMessage[]>` + `streaming: signal<boolean>`.
- **Reactive Forms with signals** — `FormControl<string>` for the compose textarea; subscribed via `takeUntilDestroyed` + `valueChanges`.
- **CSS-driven layout** — sticky compose box, scroll-y message list. No third-party "chat UI" library.
- **i18n + RTL** — every visible string under `assistant.*` namespace in `libs/i18n/src/lib/i18n/{en,ar}.json`; Material handles RTL layout flipping; bubble alignment uses `inset-inline-start` / `inset-inline-end` so user/assistant sides flip automatically.

## Acceptance criteria (rolled up from spec)

- [ ] Anonymous user lands on `/assistant`, types a question, hits Enter, sees the reply stream in token-by-token.
- [ ] At least one citation chip renders below the assistant message and links into either `/knowledge-center/resources/<slug>` or `/knowledge-maps/<id>?node=<id>`.
- [ ] User lands on `/assistant?q=What+is+CCE` and the question auto-sends; `?q=` is removed from the URL after sending.
- [ ] Cancel during streaming halts mid-token, retains the partial reply, returns focus to the compose textarea.
- [ ] Retry on a failed message replaces the failed reply.
- [ ] Regenerate on a successful message rolls back and re-streams.
- [ ] Clear-thread button confirms (when non-empty) then wipes.
- [ ] All flows work in both EN and AR with proper RTL bubble alignment.
- [ ] Backend integration test asserts Content-Type=`text/event-stream` and ≥5 `text` events + ≥1 `citation` event + 1 `done` event in order.
- [ ] axe-core reports zero critical/serious findings on `/assistant`.
- [ ] `web-portal-v0.4.0` ships; CHANGELOG entry created; ADRs 0049 + 0050 land.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| `fetch` + `ReadableStream` parsing of SSE is fiddly (chunks split mid-event) | `lib/sse-client.ts` is written test-first with cases for split-mid-event, malformed JSON, and abort-mid-stream. |
| `effect()` for auto-scroll fires too eagerly during streaming and feels jittery | Auto-scroll only triggers on **new message** (length change), not on content updates. Streaming chunks update content without re-scrolling — the bubble grows in place at the bottom of the view because the compose box is sticky. |
| `AbortController` doesn't propagate cleanly through `fetch` + `ReadableStream` on all browsers | Tested in `lib/sse-client.spec.ts`; the wrapper catches `AbortError` and rethrows as a sentinel value the store recognises. |
| Backend SSE handler stalls if the client closes mid-stream and `ct` isn't honoured | Stub uses `Task.Delay(ms, ct)` between chunks so cancellation aborts before the next yield. Integration test covers client-cancel mid-stream. |
| Stub citation references a seeded row that doesn't exist (e.g. dev DB never seeded) | Stub queries the DB for the count and short-circuits to no-citations when empty rather than failing the stream. |
| Real LLM swap-in later requires changing the response wire format | Solved by-design — SSE events are typed (`text`, `citation`, `done`, `error`) and the LLM client implements `IAsyncEnumerable<SseEvent>`. The wire shape is decoupled from the LLM SDK. |

## Out of scope (deferred to v0.x or Sub-N+1)

- **Real LLM integration** — stub fake-streams. Real client lands later.
- **Server-persisted threads** — in-memory only.
- **Markdown rendering of replies** — plain text. Stub doesn't emit markdown anyway.
- **Code blocks / math / image attachments** — defer.
- **Multi-thread management UI** — single thread per session.
- **Conversation sharing / export** — defer.
- **Voice / TTS** — out of scope.
