# CCE Sub-Project 10b — Deployment automation — Implementation Plan (Master)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan phase-by-phase. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Take the four production Docker images plus the `CCE.Seeder` console shipped in Sub-10a and turn them into a one-command deployable system on a single Windows Server 2022 host. Migrations run automatically as a sidecar; secrets stay outside the repo as `.env.prod` on the host; smoke probes verify the deploy; rollback is a single PowerShell call. Sub-10c handles AD federation, multi-env, backup automation, TLS / DNS / LB validation against IDD v1.2, and production secret provisioning.

**Architecture:** A new `cce-migrator` multistage Docker image wraps the existing `CCE.Seeder` console (extended with `--migrate` / `--seed-reference` flags backed by EF Core's `Database.MigrateAsync()`). Compose v2 file (`docker-compose.prod.yml`) wires all 5 images (4 apps + migrator) with `depends_on: { condition: service_completed_successfully }` so migrator runs to completion before APIs start. CI extends Sub-10a's `docker-build` job to push to `ghcr.io/<owner>/cce-<image>` with `:<sha>` / `:sha-<short>` / `:latest` tags on `main` (plus `:<release-tag>` on `v*` pushes). Operator deploys via `deploy/deploy.ps1`, rolls back via `deploy/rollback.ps1` (which atomically swaps `CCE_IMAGE_TAG` in `.env.prod` and re-runs deploy). `deploy-history.tsv` records every deploy + rollback. A new `deploy-smoke.yml` Windows-runner workflow simulates the full deploy → rollback cycle end-to-end.

**Tech Stack:** Existing — .NET 8 + EF Core 8 (migrations) + `CCE.Seeder` console; Docker Compose v2; GitHub Actions; `docker/build-push-action@v6` with GHA layer cache. New — `docker/login-action@v3` for ghcr.io push; PowerShell 7+ for `deploy.ps1` / `rollback.ps1` / `smoke.ps1`; ghcr.io as image registry; Testcontainers `Microsoft.SqlServer` for migration unit tests.

**Spec reference:** [`../specs/2026-05-03-sub-10b-design.md`](../specs/2026-05-03-sub-10b-design.md) — 5 scope decisions, 3-phase plan, ~14 tasks.

## Global conventions

### Working directory

All paths relative to repo root `/Users/m/CCE/`. Backend commands run from `backend/`; PowerShell scripts target the `deploy/` directory at repo root. The plan calls out `cd <dir>` where needed.

### Git workflow

- One commit per task (atomic, reviewable history).
- Conventional commit format: `<type>(<scope>): <subject>`. Scopes for Sub-10b: `feat(seeder)`, `feat(infrastructure)`, `chore(docker)`, `chore(ci)`, `chore(deploy)`, `docs(sub-10b)`, `docs(runbook)`.
- Commit with `git -c commit.gpgsign=false commit -m "..."` if signing is set up. No `--no-verify`.
- Gitleaks pre-commit hook stays active.

### TDD discipline (per ADR-0007 + Sub-6 through Sub-10a pattern)

**Strict TDD** for:
- `CCE.Seeder` flag parsing (`SeederFlagParsingTests`).
- `CCE.Seeder` migrate-mode behaviour (`SeederMigrateModeTests` against Testcontainers SQL).
- `CCE.Seeder` migrate + seed-reference idempotency (`SeederMigrateSeedReferenceTests`).

**Test-after** for:
- `CCE.Seeder/Dockerfile` — verified by CI build success + smoke check (`docker compose run --rm migrator --migrate` against an inline SQL container).
- `deploy/deploy.ps1` / `deploy/rollback.ps1` / `deploy/smoke.ps1` — verified by Phase 02's `deploy-smoke.yml` Windows-runner workflow.
- Compose-file edits — verified by CI smoke target build + deploy-smoke workflow.
- `docker-compose.prod.deploy.yml` strict-env override — verified by a Phase 01 step that runs `docker compose ... config` with required keys missing and asserts non-zero exit.

### Patterns inherited from Sub-10a

- **Multistage Dockerfile:** `mcr.microsoft.com/dotnet/sdk:8.0` build stage → `mcr.microsoft.com/dotnet/aspnet:8.0` runtime stage. Copy `Directory.Packages.props` + `Directory.Build.props` + `NuGet.config*` first for restore-cache layering. Copy each `csproj` before `src/`. Copy `permissions.yaml` before the rest of `src/` (CCE.Domain source generator reads it from backend root). Non-root `app` user (uid 1654 — comes with `aspnet:8.0`). `HEALTHCHECK` not applicable to migrator (it exits).
- **CI `docker/build-push-action@v6`:** `cache-from: type=gha,scope=<image>` + `cache-to: type=gha,mode=max,scope=<image>` per image to share build layers across runs.
- **Smoke probes:** backend `curl -fsS http://localhost:<port>/health` waits up to 30 sec; frontend `curl -fsS http://localhost:<port>/ | grep "<html"`. Sub-10a's existing probes in `ci.yml` extend in Phase 01 to include the migrator.
- **Conditional push gate:** PR builds stay `push: false`; `main` and `v*` push to ghcr.io.

### Test running

Backend (Application + Infrastructure unit tests, no external services for non-migration tests):
```bash
cd backend && dotnet test tests/CCE.Application.Tests/ tests/CCE.Infrastructure.Tests/
```

Backend migration tests (require Docker for Testcontainers SQL Server):
```bash
cd backend && dotnet test tests/CCE.Infrastructure.Tests/ --filter "FullyQualifiedName~Migration"
```

Build:
```bash
cd backend && dotnet build
```

Docker builds (smoke):
```bash
docker build -t cce-migrator:dev -f backend/src/CCE.Seeder/Dockerfile backend/
```

Compose smoke (post-Phase 01):
```bash
# Local build target (no ghcr.io required)
docker compose -f docker-compose.prod.yml -f docker-compose.build.yml up --build -d
curl -fsS http://localhost:5001/health
docker compose -f docker-compose.prod.yml -f docker-compose.build.yml down
```

Deploy script smoke (post-Phase 01, on Windows host):
```powershell
.\deploy\smoke.ps1 -Timeout 60
```

- All new files have at least one xunit test where they represent testable behaviour. Dockerfiles + compose-file edits + PowerShell scripts are verified via CI build + smoke probes + the Phase 02 `deploy-smoke.yml` workflow instead.
- Backend: `dotnet build` and `dotnet test tests/CCE.Application.Tests/ tests/CCE.Infrastructure.Tests/` pass locally (existing baseline + new Sub-10b tests).
- New env-vars documented in `.env.prod.example` (Phase 01) — single source of truth.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| `Database.MigrateAsync()` deadlocks against an existing app pool holding connections | Migrator runs to completion before APIs start (`depends_on: service_completed_successfully`). The script path runs migrator as a one-shot before bringing up APIs, so the app processes never have open connections during migration. |
| Testcontainers SQL Server is slow on CI runners | The migration test suite is filtered with `--filter "FullyQualifiedName~Migration"` and run as a separate step. Other Infrastructure.Tests stay fast. CI reuses the SQL container across migration tests via xUnit class fixtures. |
| ghcr.io rate-limits anonymous pulls | Operator sets `CCE_GHCR_TOKEN` in `.env.prod`; `deploy.ps1` runs `docker login` if present. CI uses `GITHUB_TOKEN` (no rate limit for the actor's own org). |
| Operator typos `CCE_IMAGE_TAG`, gets a missing-image error after step 6 | `deploy.ps1` step 5 (pull) runs before any state change; `docker compose pull` on a missing tag aborts with a clear error. Print the resolved tag at step 2 for confirmation. |
| Forward-only-migration discipline gets violated, schema drift after rollback | `docs/runbooks/migrations.md` documents the rules (additive only; deprecation across releases; no destructive defaults). PR review enforces. The escape hatch (separate destructive-release plan) is documented. |
| `.env.prod.example` accidentally drifts from real key set | Phase 01 includes a CI step that asserts `.env.prod.example` parses, contains every required key documented in the spec, and is referenced from `deploy/README.md`. |
| `service_completed_successfully` not honoured by older Docker Compose | Plan pins Docker Compose v2.20+ (the spec writes `docker compose` not `docker-compose`). `deploy.ps1` step 3 (pre-flight) checks `docker compose version` and aborts on incompatible. |
| Windows path separators in `env_file:` resolved by Docker Linux engine | The compose runs Linux containers on Windows host, but the `env_file` directive is read by the Docker client on the host before being passed to the daemon. Native Windows paths (e.g. `C:\ProgramData\CCE\.env.prod`) work correctly with Docker Desktop on Windows. `deploy.ps1` exports the path before invoking compose. |
| `--no-deps` semantics differ between `docker compose run` and `docker compose up` | Spec calls for `--no-deps` on both step 6 (`run`) and step 7 (`up`). Both flags exist in compose v2 and behave consistently (skip dependency resolution). Phase 02's `deploy-smoke.yml` exercises both paths. |

## Out of scope (deferred to Sub-10c or beyond)

- **TLS termination / reverse proxy / production DNS hostnames** (CCE-ext / CCE-admin-Panel / api.CCE / Api.CCE-admin-Panel) — Sub-10c.
- **AD federation against `cce.local` (389/636)** — Sub-10c.
- **Multi-environment promotion** (test → pre-prod → prod → DR) — Sub-10c.
- **Backup automation, DB restore procedure, schema rollback** — Sub-10c.
- **Production Sentry DSN provisioning + secret rotation** — Sub-10c.
- **Auto-rollback on smoke-probe failure** — Sub-10c+. Sub-10b is operator-driven only.
- **Multi-host orchestration / clustering** — Sub-10c+.
- **Windows-native rebuild** (IIS hosting + Windows Service) — explicitly rejected during brainstorm.
- **Vault / Azure Key Vault / AWS Secrets Manager** — `.env.prod` on host with NTFS ACLs is the 10b answer.
