**Goal:** Replace the Sub-6 Phase 9 skeleton at `/interactive-city` with a single-page scenario-builder workbench. Users pick technologies from a catalog, see live totals (carbon impact, cost) recompute on every toggle, run the scenario against the server for an authoritative result + localized summary, and (when authenticated) save named scenarios for later reload. Backend unchanged â€” all endpoints already shipped in Sub-4.

**Architecture:** All new surface lives under `apps/web-portal/src/app/features/interactive-city/`. A single `ScenarioBuilderStore` (signals-first) owns the editable scenario state (city type, target year, name, selected technology IDs), the catalog cache, the saved-scenarios list, and network flags. Sub-components consume signals and emit actions; no parent prop drilling. `liveTotals` is a pure client-side computed signal (sum over selected technologies) â€” Run posts the same configuration to the server for authoritative numbers + a localized summary string. URL captures `?city=&year=&t=&name=` for deep-linking. Saved-scenarios drawer is a CSS-driven right-rail-on-desktop / bottom-sheet-on-mobile, matching `NodeDetailPanelComponent` from Sub-7.

**Tech Stack:** Angular 19 standalone, signals-first, Reactive Forms, Material 18, ngx-translate ar/en + RTL, Jest 30 (unit), Playwright + `@axe-core/playwright` (E2E + a11y). All in `frontend/package.json` â€” no new heavy deps.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Save fails silently if BFF cookie expires mid-session | Save handler treats `401` as "kick off `signIn()` and re-attempt on resume"; happy path is unchanged for valid sessions. |
| Catalog grows past ~30 technologies and the catalog list becomes unwieldy | Search box is debounced; group-by-category collapses sections; pagination is **not** introduced in v0.1.0 (YAGNI â€” backend currently returns 4 technologies). |
| URL state grows past ~2KB if a user picks many technologies | URL stores only the GUID list; 100 GUIDs â‰ˆ 3.6KB which is below browser limits. If we ever cross 50 selections, we'd switch to a server-side scenario-id ref â€” not in v0.1.0. |
| `dirty` signal flickers during URL hydration and triggers a confirm dialog on every load | Hydration sets a `hydrating` flag; `dirty` ignores changes during the hydrate window. Tested explicitly. |
| Load-while-running edge case | Run button disabled while `running()`; load action also disabled while `running()`. UI guards both ends. |
