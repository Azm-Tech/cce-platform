# CCE Sub-Project 10c — Production infra + DR — Implementation Plan (Master)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan phase-by-phase. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Take Sub-10b's deployable system from "works on one host with one env-file" to "operationally complete on the IDD-specified target hardware". AD federation via Keycloak LDAP; IDD hostnames over TLS via IIS reverse proxy; backup automation + restore; multi-environment promotion (test → preprod → prod → DR); secret-rotation procedure; auto-rollback on smoke failure; production Sentry DSN provisioning; documented DR rebuild from backups.

**Architecture:** `deploy.ps1` gains an `-Environment` switch resolving to `.env.{test,preprod,prod,dr}` env-files; per-env deploy-history audit trails. Identity rides Keycloak's User Federation provider against AD (`ldaps://cce.local:636`, read-only); group → role mapping baked into a committed realm-import JSON. Network: IIS sites (port 443, SNI by host header) terminate TLS and reverse-proxy via ARR to the 4 backend container ports (Sub-10b unchanged). Backup: Ola Hallengren maintenance solution + Windows Task Scheduler + hourly off-host robocopy sync. DR: cold standby host, RTO ~hours, RPO ≤15 minutes, promotion is a documented 8-step runbook.

**Tech Stack:** Existing — PowerShell 7+, Docker Compose v2, Keycloak (existing realm `cce`), SQL Server, Windows Server 2022, GitHub Actions, IIS + URL Rewrite + ARR. New — Ola Hallengren SQL Server Maintenance Solution v2024.x; Windows Task Scheduler (`schtasks` XML import); Testcontainers OpenLDAP for federation tests; xUnit collection-fixture pattern for SQL backup-restore tests.

**Spec reference:** [`../specs/2026-05-04-sub-10c-design.md`](../specs/2026-05-04-sub-10c-design.md) — 7 scope decisions, 6-phase plan, ~25 tasks.

## Global conventions

### Working directory

All paths relative to repo root `/Users/m/CCE/`. Backend commands run from `backend/`; PowerShell scripts target the `deploy/` directory at repo root; infra scripts target `infra/` at repo root. The plan calls out `cd <dir>` where needed.

### Git workflow

- One commit per task (atomic, reviewable history).
- Conventional commit format: `<type>(<scope>): <subject>`. Scopes for Sub-10c: `feat(deploy)`, `feat(infra)`, `feat(api-common)`, `chore(ci)`, `chore(env)`, `docs(sub-10c)`, `docs(runbook)`, `docs(adr)`, `test(infrastructure)`.
- Commit with `git -C /Users/m/CCE -c commit.gpgsign=false commit -m "..."`. No `--no-verify`.
- Gitleaks pre-commit hook stays active.

### TDD discipline (per ADR-0007 + Sub-6 through Sub-10b pattern)

**Strict TDD** for:
- `validate-env.ps1` canary patterns (Phase 00).
- `deploy.ps1` `-Environment` resolution + per-env history file (Phase 00 — extension of existing tests; assert via shell-callable side effects).
- `KeycloakLdapFederationTests` (Phase 01) — Testcontainers Keycloak + OpenLDAP fixture.
- `RestoreFromBackupTests` (Phase 03) — Testcontainers SQL backup-chain round-trip.
- `LoggingExtensionsSentryTests` (Phase 04) — env + release tag flow into Sentry options.

**Test-after** for:
- IIS provisioning scripts (Phase 02) — verified via deploy-smoke.yml on Windows runners.
- Ola Hallengren install.sql (Phase 03) — verified by integration test against Testcontainers SQL.
- DR promotion runbook (Phase 05) — verified by ops drill, not CI.

### Patterns inherited from Sub-10a/10b

- **Per-env file pattern** — extends Sub-10b's single `.env.prod` to N envs, same NTFS-locked semantics.
- **`deploy.ps1` 10-step idempotent flow** — preserves the exact step numbering; Step 2 grows to call `validate-env.ps1`; Step 8 grows the auto-rollback path; Step 9's history append goes per-env.
- **Multistage Dockerfiles** — unchanged. Sub-10c adds no new images.
- **Conventional commits** — same scope vocabulary plus `feat(infra)`.
- **xUnit collection fixtures** — `MigratorFixture` from Sub-10b is the template for `KeycloakFixture` (Phase 01) and reused for `MigratorFixture` extension (Phase 03 backup tests).

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Sub-10b's `deploy.ps1` regresses when `-Environment` parameter is added | Default value `prod` preserves backward compat; existing call sites (deploy-smoke.yml) still work without modification. New `-Environment` is opt-in. Test: existing `deploy-smoke.yml` passes unchanged after Phase 00. |
| LDAP test container takes too long on CI | Testcontainers OpenLDAP boots in ~5 seconds (much faster than SQL Server's ~30s); shared via xUnit collection fixture across all federation tests. Worst case: gate the test class behind `[Trait("Category", "Slow")]` and exclude from default `dotnet test`. |
| IIS site provisioning leaves partial state on script failure | `Configure-IISSites.ps1` wraps each site creation in a try/catch and rolls back via `Remove-IISSite` on any per-site failure. Idempotent — re-running converges. |
| Ola Hallengren install.sql diverges from upstream over time | Pin to v2024.x with a checksum header; document the upgrade procedure (download, diff, re-commit). Don't auto-update. |
| `Restore-FromBackup.ps1` overwrites live DB by accident | Refuses to restore over `CCE` DB unless `-Force` is passed; default target is `CCE_restored`. The tests use `-TargetDb CCE_restoretest`. |
| Auto-rollback infinite loop | `-Recursive` switch (set by `rollback.ps1` only) suppresses auto-rollback in the nested `deploy.ps1` call. Outer fails non-zero with "manual intervention required". Tested in deploy-smoke.yml. |
| `validate-env.ps1` false positives blocking valid deploys | Canary patterns are exact-string matches against placeholder values (`<set-me>`, `<github-org-or-user>`, etc.) and a curated leaked-secret list (e.g., AWS docs canary `AKIAIOSFODNN7EXAMPLE`). Real-world secrets won't match unless someone literally pasted documentation. |
| DR-host promotion goes wrong during a real incident | Quarterly drill on the DR host validates the runbook. `Restore-FromBackup.ps1` test covers the highest-risk step (DB restore). DNS swap is operator-managed; documented step-by-step. |
| Per-env `deploy-history-${env}.tsv` parsed by `rollback.ps1` differs from prod history file location | `rollback.ps1` already takes `-EnvFile` (Sub-10b); Phase 00 extends it to derive the correct history file from the env-file's filename suffix. Single source of truth: env-file path → env name → history file. |

## Out of scope (deferred to Sub-10d or beyond)

- **Active-passive HA** with SQL Server log-shipping or Always On Availability Groups.
- **Active-active load-balanced topology** (multi-host).
- **Vault / Azure Key Vault / AWS Secrets Manager** graduation.
- **SPNEGO / Kerberos SSO** for AD-joined clients.
- **Federation against AD FS or Azure AD** (raw LDAP only per IDD).
- **Cert auto-provisioning** (operator picks AD CS / win-acme / manual import).
- **DNS auto-provisioning**.
- **WAF / IP allowlisting**.
- **Automated Sentry org provisioning** — operator does the click-path.
- **Automated DR drills** — quarterly drill is a runbook checklist, not a CI workflow.
- **Multi-tenant / clustering**.
- **AD writes from app** — federation stays read-only.
- **Backup encryption at rest** — can be enabled via Ola's `@EncryptionAlgorithm` parameter when IDD requires it.
