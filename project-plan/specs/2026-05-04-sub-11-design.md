# Sub-Project 11 — Entra ID migration — Design Spec

**Date:** 2026-05-04
**Status:** Approved
**Predecessor:** [Sub-10c Production infra + DR](2026-05-04-sub-10c-design.md)
**Target release:** `entra-id-v1.0.0`
**Decomposition note:** Sub-11 is the post-Sub-10 identity migration. Replaces Keycloak (wired across Sub-1 through Sub-9 + Sub-10c Phase 01) with Microsoft Entra ID (formerly Azure AD) as the IdP for the entire CCE platform. Synced from on-prem AD via Entra ID Connect. CCE backend writes to Entra ID via Microsoft Graph for self-service registration.

---

## Goal

Replace Keycloak with Microsoft Entra ID as CCE's single identity provider. Multi-tenant configuration accepts both CCE-staff users (synced from `cce.local` AD via Entra ID Connect) and external partner-org users (signing in from their own Entra ID tenants). CCE backend creates users via Microsoft Graph for self-service registration. Sign-in pages branded for the CCE tenant. Conditional Access enforces MFA. Big-bang cutover swaps all 4 envs in a single maintenance window.

## Non-goals

- **AD FS** — explicitly rejected at scope-decision time. Microsoft is deprecating AD FS in favor of Entra ID; on-prem AD FS would entail server-role provisioning we don't need.
- **B2C consumer accounts** — rejected. Multi-tenant covers partner-org access; CCE is not a public consumer SaaS.
- **Single-tenant** — rejected. CCE serves users from partner organizations who have their own Entra ID; multi-tenant is the right shape.
- **Programmatic Conditional Access policy provisioning** — admin-driven via Entra ID portal. CA policies are operator-managed; CCE is policy-agnostic.
- **Cross-tenant entitlements management** — each tenant's admin assigns CCE app roles to their own users; CCE doesn't dictate cross-tenant role assignment.
- **Distributed token cache** — in-memory cache (`Microsoft.Identity.Web` default) is sufficient at single-host scale; Redis-backed cache is a Sub-11+ option if multi-host adoption arrives.
- **Branding for partner tenants** — Entra ID security boundary; CCE can only brand its own tenant's sign-in page. Partners see their own org branding.
- **SSO with non-Microsoft IdPs** — Sub-11 is Entra-only; partners federate via their own tenant's IdP integrations.
- **Service-to-service auth between APIs** — backend network policies + per-API JWT bearer remain unchanged; this isn't an Entra ID app-to-app concern.
- **Migration of existing Keycloak MFA registrations** — users re-enroll MFA under Conditional Access policy on first sign-in.
- **AD FS server provisioning** — out of scope by virtue of choosing Entra ID over AD FS.

## Scope decisions (locked during brainstorm)

| # | Decision | Notes |
|---|---|---|
| 1 | **Multi-tenant Entra ID** | Two user populations: synced internal (CCE-staff via Entra ID Connect) + external partners (own tenants). Issuer accepts `login.microsoftonline.com/<any-tenant>/v2.0`. |
| 2 | **Microsoft.Identity.Web on top of JwtBearer + OpenIdConnect** | Multi-tenant + Graph integration is exactly what `Microsoft.Identity.Web` is built for. Existing 8.0.10 packages stay; new package is additive. |
| 3 | **Entra ID app roles (not security groups)** | App role names match today's Keycloak roles (`cce-admin`, `cce-editor`, `cce-reviewer`, `cce-expert`, `cce-user`). `permissions.yaml` consumes them unchanged. |
| 4 | **Conditional Access enforces MFA** | App stays MFA-agnostic. CCE-tenant CA policy enforces for staff; partner tenants enforce per their org policy. |
| 5 | **CCE backend creates users via Microsoft Graph** | Option (b): app-only `User.ReadWrite.All` permission. Existing `register.page.ts` form unchanged; backend swaps Keycloak Admin API for Graph. |
| 6 | **Big-bang cutover, all 4 envs same maintenance window** | Coordinated freeze + per-env deploys in sequence (test → preprod → prod → dr) within one window. Rollback is all-envs-back simultaneously. |
| 7 | **Lazy resolve UPN → objectId on first sign-in** | New `EntraIdObjectId` column on `Users`. Resolver runs in `OnTokenValidated` (BFF) + `IClaimsTransformation` (API). |
| 8 | **Recorded HTTP fixtures (WireMock.Net)** | Hermetic; no real Entra ID at test time. Operator records once against dev tenant; PII scrubbed; commits JSON fixtures. |

### API JWT validation (`CceJwtAuthRegistration.AddCceJwtAuth`)

```csharp
public static IServiceCollection AddCceJwtAuth(this IServiceCollection services, IConfiguration config)
{
    services
        .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
        .AddMicrosoftIdentityWebApi(config, configSectionName: "EntraId");

    services.Configure<JwtBearerOptions>(JwtBearerDefaults.AuthenticationScheme, opts =>
    {
        opts.TokenValidationParameters.ValidateIssuer = true;
        opts.TokenValidationParameters.IssuerValidator = (issuer, _, _) =>
            issuer.StartsWith("https://login.microsoftonline.com/", StringComparison.OrdinalIgnoreCase)
            && issuer.EndsWith("/v2.0", StringComparison.OrdinalIgnoreCase)
                ? issuer
                : throw new SecurityTokenInvalidIssuerException($"Issuer {issuer} not recognized.");
        opts.TokenValidationParameters.RoleClaimType = "roles";
    });

    services.AddSingleton<IClaimsTransformation, RoleToPermissionClaimsTransformer>();
    return services;
}
```

### BFF wiring (`BffRegistration.AddCceBff`)

```csharp
public static IServiceCollection AddCceBff(this IServiceCollection services, IConfiguration config)
{
    services
        .AddAuthentication(OpenIdConnectDefaults.AuthenticationScheme)
        .AddMicrosoftIdentityWebApp(config, configSectionName: "EntraId")
        .EnableTokenAcquisitionToCallDownstreamApi(new[] { "User.Read" })
        .AddInMemoryTokenCaches();

    services.Configure<OpenIdConnectOptions>(OpenIdConnectDefaults.AuthenticationScheme, opts =>
    {
        opts.Events.OnTokenValidated = async ctx =>
        {
            var resolver = ctx.HttpContext.RequestServices.GetRequiredService<EntraIdUserResolver>();
            await resolver.EnsureLinkedAsync(ctx.Principal!);
        };
    });

    services.AddCookie(opts =>
    {
        opts.Cookie.Name = "cce-bff-session";
        opts.Cookie.HttpOnly = true;
        opts.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        opts.Cookie.SameSite = SameSiteMode.Lax;
        opts.SlidingExpiration = true;
        opts.ExpireTimeSpan = TimeSpan.FromHours(8);
    });
    return services;
}
```

`Microsoft.Identity.Web.UI` provides `/signin`, `/signin-oidc`, `/signout`, `/signout-oidc` endpoints automatically.

### `EntraIdUserResolver`

Lazy-link UPN → Entra ID `objectId` on first sign-in. Idempotent. Concurrency-safe via filtered unique index.

```csharp
public async Task EnsureLinkedAsync(ClaimsPrincipal principal)
{
    var oidClaim = principal.FindFirstValue("oid");
    if (!Guid.TryParse(oidClaim, out var objectId)) return;

    if (await _db.Users.AnyAsync(u => u.EntraIdObjectId == objectId)) return;

    var upn = principal.FindFirstValue("preferred_username") ?? principal.FindFirstValue(ClaimTypes.Upn);
    var user = await _db.Users.FirstOrDefaultAsync(u => u.Email == upn || u.UserPrincipalName == upn);
    if (user is null)
    {
        user = User.Create(objectId, upn!, principal.Identity!.Name ?? upn!);
        _db.Users.Add(user);
    }
    else
    {
        user.LinkEntraIdObjectId(objectId);
    }
    await _db.SaveChangesAsync();
}
```

External partner edge case: no matching CCE row → create stub User with `objectId` + `Email` + `DisplayName` only. Other fields populated when user completes profile in CCE.

## Sign-in flow

```
User clicks "Sign in"
  → Browser navigates to https://CCE-ext/signin
  → BFF (Microsoft.Identity.Web) redirects to https://login.microsoftonline.com/common/oauth2/v2.0/authorize
  → Entra ID shows CCE-branded sign-in page (CCE tenant) or partner-org's branding (partner tenant)
  → Conditional Access evaluates (MFA prompt if required)
  → Browser redirects to https://CCE-ext/signin-oidc?code=...
  → BFF exchanges code for tokens; sets cce-bff-session cookie
  → EntraIdUserResolver.EnsureLinkedAsync runs (lazy DB link)
  → RoleToPermissionClaimsTransformer expands 'roles' → 'permission' claims
  → Browser redirects to / (landing page)
```

## Cutover (big-bang)

12-step maintenance-window procedure documented in `docs/runbooks/entra-id-cutover.md`:

1. Freeze writes (status page → maintenance).
2. Snapshot DBs (extra full backup outside schedule).
3. Test env: `deploy.ps1 -Environment test` with new image tag (migrator runs `AddEntraIdObjectIdToUser`).
4. Verify test (smoke probes + manual sign-in: AD-synced + partner user; MFA fires).
5. Preprod env (same as 3-4).
6. Verify preprod.
7. Prod env (same as 3-4).
8. Verify prod (smoke probes + monitor Sentry for sign-in error spikes).
9. DR env (deploy but don't flip DNS).
10. External-client smoke (production hostnames; sign-in + known-good API call).
11. Lift the freeze.
12. Decommission Keycloak (stop service; uninstall after 7 days of stable Entra ID operation).

**Rollback within window:** `rollback.ps1 -ToTag <pre-Sub-11-tag> -Environment <env>`. Forward-only-friendly migration means old Keycloak image runs against new schema unchanged.

**Rollback after window:** same script, but if Keycloak's been uninstalled (after 7-day stability), restore from backup of Keycloak's `data/` directory before rolling back.

**Maintenance window:** ~2-4 hours total.
