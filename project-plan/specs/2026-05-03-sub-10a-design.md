# Sub-Project 10a — App productionization — Design Spec

**Date:** 2026-05-03
**Status:** Approved
**Predecessor:** [Sub-9 Smart Assistant](2026-05-02-sub-9-design.md)
**Target release:** `app-v1.0.0`
**Decomposition note:** Sub-10 splits into 10a / 10b / 10c. 10a is app-layer only (no infra/network/IIS scripting). 10b adds deployment automation against one environment. 10c covers AD federation, multi-env, backup, DNS/LB validation against IDD v1.2.

---

## Goal

Take the four shipped applications (`Api.External`, `Api.Internal`, `web-portal`, `admin-cms`) from "passes tests on a developer laptop" to "production-quality builds with real observability and a real LLM". No infrastructure decisions yet — that's 10b. Sub-10a fills the three Sub-7/8/9 polish items deferred by name (Lighthouse audit, axe-core CI gates, real LLM client) and adds the production-build container images they all need anyway.

## Non-goals

- **Deployment scripts / IIS hosting / Windows Server topology** — Sub-10b.
- **AD federation, DNS / LB validation against IDD v1.2** — Sub-10c.
- **Multi-env config (test / pre-prod / prod / DR)** — Sub-10c.
- **Backup automation, secret rotation** — Sub-10c.
- **Embedding-based RAG / vector store** — citation search uses lightweight keyword overlap; full embeddings are a future sub-project.
- **Sentry production DSN provisioning** — Sub-10a wires the sink with an env-var gate; the actual production DSN lands in 10c.
- **Live-LLM tests in CI** — CI keeps using the stub. The real client is exercised manually + via a single integration test that hits a recorded fixture.
- **Promotion of `ConfirmDialogComponent` to `ui-kit`** (Sub-9 polish backlog) — out of scope for 10a.
- **Markdown rendering of assistant replies** (Sub-9 polish backlog) — out of scope for 10a.

### Container topology

Each app is built as a single multistage Docker image. Backend runtime is `mcr.microsoft.com/dotnet/aspnet:8.0` (Linux). Frontend runtime is `nginx:alpine` serving the Nx production build with an SPA-fallback rule.

**Why Linux containers and not Windows-native?** The IDD prescribes Windows Server 2022 hosts but doesn't dictate Linux-vs-Windows containers. Linux .NET 8 images are smaller, build faster, and Windows Server 2022 supports running Linux containers. Sub-10b decides whether prod actually deploys the Linux containers or rebuilds for Windows-native — that decision is gated on operations preferences, not on Sub-10a deliverables.

The `docker-compose.prod.yml` is a **smoke-test target** for Sub-10a, not a deployment artifact. It wires the four images so `docker compose up` brings them all online against developer-supplied env-vars, but it doesn't include SQL Server / Redis / Keycloak (those come from `docker-compose.yml` in dev or from the IDD-prescribed prod hosts in 10b/10c).

## Data contracts

### Configuration shape

`appsettings.json` (committed, defaults to stub provider):

```json
{
  "Assistant": {
    "Provider": "stub",
    "Anthropic": {
      "Model": "claude-sonnet-4-5-20250929",
      "MaxTokens": 1024,
      "Temperature": 0.3
    }
  },
  "Serilog": {
    "MinimumLevel": "Information",
    "FileSink": {
      "Enabled": true,
      "Path": "logs/cce-.log",
      "RetainedDays": 7
    }
  }
}
```

Env-var overrides applied at host build via `AddEnvironmentVariables()` (already in `Program.cs`):

| Env-var | Maps to | Notes |
|---|---|---|
| `ASSISTANT_PROVIDER` | `Assistant:Provider` | `stub` (default) or `anthropic` |
| `ANTHROPIC_API_KEY` | injected via separate config key, never in appsettings | absent → real client logs warn + falls back to stub |
| `SENTRY_DSN` | `Serilog:Sentry:Dsn` | absent → Sentry sink is no-op |
| `LOG_LEVEL` | `Serilog:MinimumLevel` | default Information |

### `AnthropicOptions` C# record

```cs
public sealed record AnthropicOptions
{
    public string Model { get; init; } = "claude-sonnet-4-5-20250929";
    public int MaxTokens { get; init; } = 1024;
    public double Temperature { get; init; } = 0.3;
}
```

### `AssistantClientFactory` DI behaviour

Registered as `IServiceCollection.AddCceAssistantClient(IConfiguration)`:

1. Reads `Assistant:Provider`.
2. If `"anthropic"` AND `ANTHROPIC_API_KEY` is set → registers `AnthropicSmartAssistantClient` as `ISmartAssistantClient`.
3. Otherwise → registers `SmartAssistantClient` (the existing fake-streaming stub) and emits a startup `ILogger.LogInformation` line stating which provider was chosen and why.

This keeps the stub in the test/CI hot path, lets local dev opt into the real client by setting two env-vars, and never blocks startup on missing credentials in environments that don't need them.

## Components

### `LoggingExtensions.UseCceSerilog`

Host extension that:
- Reads `Serilog:MinimumLevel`, falling back to `Information`.
- Always adds the **console** sink in JSON-compact format (one line per event — `kubectl logs` / `docker logs` ready).
- Adds the **rolling-file** sink under `logs/cce-.log` with daily rolling and `Serilog:FileSink:RetainedDays` retention (default 7), gated on `Serilog:FileSink:Enabled`.
- Adds the **Sentry** sink (Sentry.Serilog package, already in `Directory.Packages.props`) gated on `SENTRY_DSN` env-var. Captures Warning+ levels.
- Enriches every log event with: `correlation_id` (read from existing `CorrelationIdMiddleware`'s `Activity.Current?.GetBaggageItem("correlation_id")`), `locale` (from `HttpContext` `Accept-Language`), `user_id` (from `HttpContext.User.FindFirst("sub")`).

### `PrometheusExtensions.UseCcePrometheus`

`WebApplication` extension that:
- Calls `app.UseHttpMetrics()` (request duration histogram + status counters per route, prom-net default labels).
- Calls `app.MapMetrics("/metrics")` with `[AllowAnonymous]`.
- Adds a small set of custom counters: `cce_assistant_streams_total{provider}`, `cce_assistant_citations_total{kind}` for assistant observability.

### `AnthropicSmartAssistantClient`

Implements `ISmartAssistantClient`. Constructor takes:
- `IOptions<AnthropicOptions>`
- `Anthropic.SDK.AnthropicClient` (registered as singleton via `Anthropic.SDK`'s DI extension)
- `ICceDbContext` (for citation search)
- `ILogger<AnthropicSmartAssistantClient>`

`StreamAsync` flow:
1. Build a system prompt: `"You are the CCE Knowledge Center assistant. Answer in {locale}. Be concise (2-4 sentences). The user's question relates to circular carbon economy topics."`
2. Pass the message history verbatim to `client.Messages.StreamClaudeMessageAsync(...)`.
3. For each `MessageStreamEvent.ContentBlockDelta` with type=`text_delta`: yield `TextEvent(delta.Text)`.
4. After the message stream completes, call `CitationSearch.FindCitationsAsync(lastUser, lastAssistantText, locale, ct)` to get up to 1 resource + 1 map-node match. Yield each as `CitationEvent`.
5. Yield `DoneEvent`.
6. On any thrown exception: log + yield `ErrorEvent("server")`. Exception type doesn't leak to clients.

### `CitationSearch`

Pure utility — takes the user's question + the model's reply text + a locale, runs a substring/token-overlap match against `Resources.TitleEn/Ar` and `KnowledgeMapNodes.NameEn/Ar`. Returns the top 1 of each kind, scored by token overlap (Jaccard-ish: intersection size / union size). If no resource/map-node has a non-zero score, returns nothing for that kind. The signature:

```cs
public interface ICitationSearch
{
    Task<IReadOnlyList<CitationDto>> FindCitationsAsync(
        string userQuestion,
        string assistantReply,
        string locale,
        CancellationToken ct);
}
```

### Production Dockerfiles

**Backend (Api.External / Api.Internal):**
- Stage 1: `mcr.microsoft.com/dotnet/sdk:8.0` — `dotnet restore` then `dotnet publish -c Release -o /app/publish --no-restore /p:UseAppHost=false`.
- Stage 2: `mcr.microsoft.com/dotnet/aspnet:8.0` — copies publish output, `EXPOSE 8080`, `ENTRYPOINT ["dotnet", "CCE.Api.External.dll"]`.
- `.dockerignore` excludes `bin/`, `obj/`, `artifacts/`, test projects.

**Frontend (web-portal / admin-cms):**
- Stage 1: `node:22-alpine` — `corepack enable && pnpm install --frozen-lockfile` then `nx build <app> --configuration=production`.
- Stage 2: `nginx:alpine` — copies `dist/apps/<app>/browser` to `/usr/share/nginx/html`, copies the per-app `nginx.conf` to `/etc/nginx/conf.d/default.conf`, `EXPOSE 8080`.

**`nginx.conf`** (per app — they're identical except for the path):
- Listen 8080.
- Gzip text/css/js/json.
- `try_files $uri $uri/ /index.html;` SPA fallback so deep links work.
- Cache headers: hashed-asset bundles get `Cache-Control: public, immutable, max-age=604800`.

## Error handling

- **Anthropic API down / rate-limited** → `AnthropicSmartAssistantClient` catches the SDK exception, logs at Error level (Sentry sink picks it up if configured), yields `ErrorEvent("server")`. Frontend already handles this status (toast + retry button on the assistant message).
- **`ANTHROPIC_API_KEY` missing when provider=anthropic** → `AssistantClientFactory` falls back to stub on startup with a Warning log. UI continues to work; `cce_assistant_streams_total{provider="stub"}` counter increments.
- **Citation search throws** (DB issue) → swallowed inside `AnthropicSmartAssistantClient`; the assistant response still streams, just without citations. Logged at Warning.
- **Sentry DSN invalid** → Sentry's own initialization handles it; the rest of the app continues with console + file sinks only.
- **Frontend Docker build fails** in CI → workflow fails the PR; no merge.
- **Lighthouse / axe-core fail thresholds** → workflow fails the PR.

## i18n + RTL

No new user-visible strings in Sub-10a — observability + LLM swap are non-visual. The system prompt to the LLM passes the user's locale (`ar` / `en`); the model itself returns the chosen language. No new i18n keys.

## Phasing (4 phases, ~18 tasks)

- **Phase 00 — Cross-cutting** (~5 tasks): add `Anthropic.SDK` to `Directory.Packages.props`; create empty `LoggingExtensions.cs` + `PrometheusExtensions.cs` skeletons in `CCE.Api.Common/Observability/`; create `AnthropicOptions` + `AssistantClientFactory` skeletons in `CCE.Infrastructure/Assistant/`; add `Assistant:` + `Serilog:` sections to both APIs' `appsettings.json` (defaults `Provider=stub`); register `AssistantClientFactory` in `Infrastructure/DependencyInjection.cs` returning the existing stub. **No behaviour change yet.**

- **Phase 01 — Production Docker images** (~5 tasks): write `Dockerfile` for `Api.External`; same for `Api.Internal`; write `Dockerfile` + `nginx.conf` for `web-portal`; same for `admin-cms`; write `docker-compose.prod.yml`; modify `ci.yml` to add the `docker-build` job. Each image must build successfully and pass the smoke probe.

- **Phase 02 — Observability + real LLM client** (~5 tasks): wire `UseCceSerilog` into both API hosts (replaces existing `app.Logger.Log...` console-only setup); wire `UseCcePrometheus` into both API hosts (`/metrics` endpoint); implement `CitationSearch` + tests; implement `AnthropicSmartAssistantClient` + tests (with mocked SDK); flip `AssistantClientFactory` to honour `Assistant:Provider` value. **At end of Phase 02, the app actually streams from Claude when the env-vars are set.**

- **Phase 03 — CI gates + close-out** (~3 tasks): add `lighthouse.yml` workflow; add `a11y.yml` workflow + `apps/web-portal-e2e/src/a11y.spec.ts`; ADRs (0051 — Anthropic SDK + RAG-lite citations, 0052 — observability stack: Serilog → console+file+Sentry, Prometheus middleware + custom counters); `docs/sub-10a-app-productionization-completion.md`; CHANGELOG entry under `app-v1.0.0`; tag `app-v1.0.0`.

## ADRs to land in Phase 03

- **ADR-0051 — Anthropic SDK + RAG-lite citations**: chosen over OpenAI / provider-agnostic OpenAI-compat REST because the assistant's UX (streaming + citations + tool-use-ready) maps cleanly onto Claude's streaming API and the project's existing tooling. RAG-lite (substring/token overlap) chosen over embeddings because the seeded data is small (~2 maps × 13 nodes + 5 resources) and the marginal recall gain from embeddings doesn't justify a vector store dependency in v1.0.
- **ADR-0052 — Observability stack: Serilog + Sentry + Prometheus**: chosen over OpenTelemetry-everything because the packages were already in `Directory.Packages.props` and each service tier (logs / errors / metrics) wants a different sink. OTel migration deferred until production scale demands it.
