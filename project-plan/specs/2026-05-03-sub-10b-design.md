# Sub-Project 10b ‚Äî Deployment automation ‚Äî Design Spec

**Date:** 2026-05-03
**Status:** Approved
**Predecessor:** [Sub-10a App productionization](2026-05-03-sub-10a-design.md)
**Target release:** `deploy-v1.0.0`
**Decomposition note:** Sub-10 splits into 10a / 10b / 10c. 10a (app productionization) shipped as `app-v1.0.0`. 10b wraps those artefacts into a deployable unit targeting one environment end-to-end. 10c covers AD federation, multi-env, backup automation, TLS / DNS / LB validation against IDD v1.2, and production secret provisioning.

---

## Goal

Take the four production Docker images and the `CCE.Seeder` console shipped in Sub-10a and turn them into a one-command deployable system on a single Windows Server 2022 host. Migrations run automatically, secrets stay outside the repo, smoke probes verify the deploy, and rollback is a single PowerShell call away. No infrastructure decisions about TLS, DNS, AD, or multi-env ‚Äî that is 10c.

## Non-goals

- **TLS termination / reverse proxy / production DNS hostnames** (CCE-ext / CCE-admin-Panel / api.CCE / Api.CCE-admin-Panel) ‚Äî Sub-10c.
- **AD federation against `cce.local` (389/636)** ‚Äî Sub-10c.
- **Multi-environment promotion** (test ‚Üí pre-prod ‚Üí prod ‚Üí DR) ‚Äî Sub-10c.
- **Backup automation, DB restore procedure, schema rollback** ‚Äî Sub-10c.
- **Production Sentry DSN provisioning + secret rotation** ‚Äî Sub-10c.
- **Auto-rollback on smoke-probe failure** ‚Äî Sub-10c+ judgment call. 10b is operator-driven only.
- **Multi-host orchestration / clustering** ‚Äî Sub-10c+.
- **Windows-native rebuild (IIS hosting + Windows Service)** ‚Äî explicitly rejected; we ship the Linux containers Sub-10a built and run them on a Windows Server 2022 host via Docker (Desktop / CE / Mirantis runtime).
- **Vault / Azure Key Vault / AWS Secrets Manager integration** ‚Äî `.env.prod` on the host with NTFS-restricted ACLs is the 10b answer. Vault graduation is a 10c+ decision.
- **A new test project** ‚Äî all unit tests land in existing projects (`CCE.Infrastructure.Tests`).

## Scope decisions (locked during brainstorm)

| # | Decision | Notes |
|---|---|---|
| 1 | **Linux containers on Windows Server 2022** | Reuse Sub-10a's 4 images verbatim. No Windows-native rebuild. |
| 2 | **Sidecar migrator runs to completion before APIs** | Compose `depends_on: { condition: service_completed_successfully }`. Configurable via `MIGRATE_ON_DEPLOY`. |
| 3 | **`.env.prod` on the host (NTFS-restricted)** | `C:\ProgramData\CCE\.env.prod`, mode-locked via `icacls`. |
| 4 | **Image-tag rollback + forward-only migrations** | Atomic, fast. Destructive migrations require a separate plan. |
| 5 | **ghcr.io + PowerShell deploy script** | `ghcr.io/<owner>/cce-<image>:<tag>`. One `deploy.ps1` does it all. |

### Tag scheme

Every CI build on `main` pushes three tags per image:

| Tag | Purpose |
|---|---|
| `:<full-git-sha>` | Immutable rollback target. |
| `:sha-<7-char>` | Human-readable rollback target (e.g. `sha-c612812`). |
| `:latest` | Convenience pointer to latest `main`. |

Releases (`v*` tags) additionally push `:<release-tag>` (e.g. `:deploy-v1.0.0`, `:app-v1.0.1`).

PR builds are `push: false` ‚Äî they build images locally and run smoke probes without pushing. This is Sub-10a's existing behaviour.

## Components

### `CCE.Seeder` ‚Äî migration mode

**New flags:**

| Flag combination | Behaviour |
|---|---|
| `--migrate` | Resolves `CceDbContext`, calls `await ctx.Database.MigrateAsync(ct)`, logs the applied migration list to stdout, exits 0. Exits non-zero on any exception. |
| `--migrate --seed-reference` | Above, then runs `ReferenceDataSeeder` + `RolesAndPermissionsSeeder` + `KnowledgeMapSeeder` (all idempotent ‚Äî existing seeders check for existing rows). Never runs `DemoDataSeeder`. |
| `--demo` | Existing dev-time behaviour ‚Äî runs all seeders including demo. Unchanged. |
| (no flag) | Existing dev-time behaviour ‚Äî runs all non-demo seeders. Unchanged. |
| `--migrate --demo` | Rejected. Exits non-zero with a clear error: "Demo data is not allowed in migration mode." |

**Idempotency:** running `--migrate --seed-reference` twice produces the same database state. Migrations are no-op when nothing's pending; seeders skip rows that already exist.

**Logging:** Serilog (already wired by `AddInfrastructure` chain) emits structured records with `app=CCE.Seeder`, `mode=migrate|seed|none`, and a per-migration line listing applied migration names.

### `CCE.Seeder/Dockerfile`

Multistage, mirror of API Dockerfiles:

```
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY Directory.Build.props Directory.Packages.props nuget.config* ./
COPY permissions.yaml ./
COPY src/ ./src/
RUN dotnet restore src/CCE.Seeder/CCE.Seeder.csproj
RUN dotnet publish src/CCE.Seeder/CCE.Seeder.csproj -c Release -o /app/publish --no-restore

FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app/publish ./
ENTRYPOINT ["dotnet", "CCE.Seeder.dll"]
```

`permissions.yaml` is required by the CCE.Domain source generator, same as the API Dockerfiles.

Entry uses `ENTRYPOINT` (not `CMD`) so compose can pass flags as `command:`.

### `docker-compose.prod.yml` ‚Äî extended

The Sub-10a `docker-compose.prod.yml` (4 apps, no infra) gains a `migrator` service and switches all 5 services from `build:` blocks to `image:` refs:

```yaml
services:
  migrator:
    image: ghcr.io/${CCE_REGISTRY_OWNER}/cce-migrator:${CCE_IMAGE_TAG:-latest}
    env_file: ${CCE_ENV_FILE}
    command: ["--migrate", "--seed-reference"]
    restart: "no"

  api-external:
    image: ghcr.io/${CCE_REGISTRY_OWNER}/cce-api-external:${CCE_IMAGE_TAG:-latest}
    env_file: ${CCE_ENV_FILE}
    depends_on:
      migrator:
        condition: service_completed_successfully
    ports: ["5001:8080"]

  api-internal:
    image: ghcr.io/${CCE_REGISTRY_OWNER}/cce-api-internal:${CCE_IMAGE_TAG:-latest}
    env_file: ${CCE_ENV_FILE}
    depends_on:
      migrator:
        condition: service_completed_successfully
    ports: ["5002:8080"]

  web-portal:
    image: ghcr.io/${CCE_REGISTRY_OWNER}/cce-web-portal:${CCE_IMAGE_TAG:-latest}
    ports: ["4200:8080"]

  admin-cms:
    image: ghcr.io/${CCE_REGISTRY_OWNER}/cce-admin-cms:${CCE_IMAGE_TAG:-latest}
    ports: ["4201:8080"]
```

`CCE_ENV_FILE` has no default ‚Äî `deploy.ps1` always exports it to the resolved env-file path before invoking compose. Direct `docker compose` use (e.g. deploy-smoke CI) sets it explicitly. Compose hard-fails if the variable is unset, surfacing operator misconfig immediately.

**Two ways the migrator runs:**
- **Direct `docker compose up`** (deploy-smoke CI workflow, debugging) ‚Äî `depends_on: service_completed_successfully` orchestrates ordering. Migrator runs to completion, then APIs start.
- **`deploy.ps1` orchestrated path** ‚Äî script invokes migrator explicitly with `docker compose run --rm migrator` to capture exit code, then brings up APIs with `--no-deps` to bypass the now-empty dependency. Both paths produce identical end state.

The Sub-10a `${KEYCLOAK_AUTHORITY:-...}`-style defaults move to a dedicated **build/CI smoke target** override (`docker-compose.build.yml`). Production runs use `docker-compose.prod.deploy.yml` which strips defaults so missing required keys hard-fail at compose-up time.

### `deploy/deploy.ps1`

Single host-side entry point. Idempotent. Returns `0` on success, non-zero on any failure.

**10-step flow** (each logs to `C:\ProgramData\CCE\logs\deploy-<UTC-stamp>.log`):

1. **Resolve env-file path.** Default `C:\ProgramData\CCE\.env.prod`, override via `-EnvFile <path>`.
2. **Validate env-file.** Parse keys, check required-key list, abort with non-zero on missing/empty. Print resolved `CCE_IMAGE_TAG` for confirmation.
3. **Pre-flight: docker reachable?** `docker info` ‚Äî abort if Docker daemon isn't responding.
4. **Pre-flight: registry login.** If `CCE_GHCR_TOKEN` set, run `docker login ghcr.io -u <owner> --password-stdin`. Otherwise trust the existing session.
5. **Pull images.** `docker compose -f docker-compose.prod.yml -f docker-compose.prod.deploy.yml --env-file <path> pull`. Aborts if any image is missing for the requested tag (catches typos in `CCE_IMAGE_TAG`).
6. **Migrator step.** Skip if `MIGRATE_ON_DEPLOY=false`. Otherwise `docker compose ... run --rm --no-deps migrator` (the migrator's `command:` from YAML drives `--migrate --seed-reference`). `docker compose run` returns the container's exit code natively. Captures migrator stdout+stderr to log; aborts if exit code ‚âÝ 0.
7. **Up the apps.** `docker compose ... up -d --no-deps --remove-orphans api-external api-internal web-portal admin-cms`. `--no-deps` bypasses the `depends_on: service_completed_successfully` directive (which is for direct `compose up` use); the script already orchestrated the migrator explicitly in step 6.
8. **Smoke probe.** Calls `deploy/smoke.ps1`. Polls `http://localhost:5001/health`, `http://localhost:5002/health`, `http://localhost:4200/`, `http://localhost:4201/`. 30 attempts √ó 2 sec backoff = 60-second window per endpoint. Asserts HTTP 200 + (for `/health`) JSON `status: "Healthy"`.
9. **Append `deploy-history.tsv`.** `<UTC-timestamp>\t<git-sha>\t<image-tag>\t<outcome>`.
10. **Print summary.** Image refs deployed, migration outcome, smoke-probe results, log file path. On any failure after step 5, print exact rollback command.

**Behaviour on failure:**
- Migrator non-zero ‚Üí script aborts before touching APIs.
- Smoke probe failure ‚Üí script does **not** auto-rollback. Logs failure, prints rollback command, exits non-zero.
- Pull failure ‚Üí script aborts before changing running state.

### `deploy/rollback.ps1`

Thin wrapper around `deploy.ps1`. Required: `-ToTag <tag>`. Optional: `-EnvFile <path>`, `-SkipMigrator`.

**Flow:**
1. Read current `.env.prod`, capture outgoing tag.
2. Atomically rewrite `.env.prod` with `CCE_IMAGE_TAG=<new-tag>` (temp-file + rename), preserving every other line.
3. Invoke `deploy.ps1 -EnvFile <path> [-SkipMigrator]`.
4. Append `rollback-<UTC-stamp>.log` to `C:\ProgramData\CCE\logs\` capturing the swap.

**`-SkipMigrator`:** rollback target's images expect at-or-older schema (forward-only discipline), so `MigrateAsync` is a no-op. Switch exists for the rare case where the operator wants to bypass even that.

## Error handling

| Failure | Detection | Behaviour |
|---|---|---|
| Required env-key missing | `deploy.ps1` step 2 | Abort before touching Docker. Print missing keys. Exit non-zero. |
| Docker daemon unreachable | `deploy.ps1` step 3 | Abort. Print `docker info` error. Exit non-zero. |
| Image pull fails (typo in tag, registry down, missing auth) | `deploy.ps1` step 5 | Abort before changing running state. Print `docker compose pull` error. Exit non-zero. |
| Migration fails | `deploy.ps1` step 6 | Abort before touching APIs. Capture migrator stdout/stderr to log. Print rollback command. Exit non-zero. |
| Smoke probe fails | `deploy.ps1` step 8 | Apps left running (operator may want to inspect logs). Print rollback command. Exit non-zero. **No auto-rollback.** |
| `rollback.ps1 -ToTag` typo | Image pull step inside `deploy.ps1` | Aborts before swapping containers. `.env.prod` already updated to bad tag ‚Äî operator must re-run `rollback.ps1` with correct tag. |
| Schema drift after rollback | Apps fail to start / `/health` returns Unhealthy | STOP. Forward-only discipline says this shouldn't happen. File incident. 10c+ scope (DB-level recovery). |
| ghcr.io rate-limit / auth failure | `deploy.ps1` step 4 or 5 | Operator sets `CCE_GHCR_TOKEN` in `.env.prod` and retries. |

## Migration discipline (forward-only)

Captured in `docs/runbooks/migrations.md`. Rules:

1. **Additive only:** add columns, add tables, add indexes. Never drop, rename in place, or change types.
2. **Deprecation across releases:** to remove a column, release N adds the replacement and dual-writes; release N+1 stops reading the old column; release N+2 drops it. Three releases minimum.
3. **No destructive defaults:** new NOT NULL columns must have a `DEFAULT` expression that handles existing rows.
4. **Online indexes where possible:** `CREATE INDEX WITH (ONLINE = ON)` against large tables.
5. **Rollback semantics:** with these rules, an old image safely runs against a newer schema. Rollback = swap image tag.
6. **Destructive case escape hatch:** if a release truly needs a destructive change, plan it as a separate release with an explicit data-migration runbook. Out of 10b scope, but the doc spells out the approval gate.
