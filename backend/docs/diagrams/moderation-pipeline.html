Content Moderation Pipeline

Two-phase automated analysis (rule filter → AI) followed by a human review queue. Async from publish; content surfaces immediately and is retracted only on violation.

Transport · MassTransit InMemory / RabbitMQ AI Provider · Groq llama-3.1-8b-instant Outbox · EF Core + SQL Server
Surface → Retract model. Content is visible immediately on publish. The consumer runs ~1–2 s later and retracts only if a rule or AI verdict fires. This is honest — we catch violations quickly after, not before they surface. Feed and search queries carry no ModerationStatus predicate; only Rejected content is evicted explicitly.

02Dispatch
EF Core Interceptor · SavingChangesAsync · Pre-commit
DomainEventDispatcher
Drains domain events from all tracked aggregates and publishes via MediatR before the SQL transaction commits. Any outbox rows added by the handlers are included in the same SaveChanges, making event delivery atomic with the aggregate write.
MediatR Handler · moderation path

MediatR Handlers · other fan-out
Feed · Notifications · SignalR

Parallel fan-out to follower feeds, notification consumers, real-time SignalR hub. Runs concurrently with moderation.
03Transport
EF Core Outbox · SQL Server
outbox_message
Message row committed atomically with the post in a single SQL transaction. No dual-write, no lost-message window — the message exists only if the post exists.
MassTransit · QueryDelay = 1 s
BusOutboxDeliveryService
Polls outbox_message every second, stamps sent_time, and relays rows to the bus. Dev → InMemory. Prod → RabbitMQ. Same code, different transport.
⚠ InMemory limitation: two API processes sharing the same DB both run BusOutboxDeliveryService. Occasional delivery to the wrong process causes silent message loss in dev. Not present with RabbitMQ.
04Consumer
MassTransit Consumer · ConcurrentMessageLimit = 5 · Retry: 2 s → 10 s → 30 s
ContentModerationConsumer
Receives ContentModerationRequestedIntegrationEvent. Phase 1 fires unconditionally; Phase 2 (AI) fires only if Phase 1 passes. Scope is created after both phases — the DB commit from step 01 has completed by then, so FindPostAsync returns the live row.

05Decision
Status Resolution · ResolveStatus(score, phase)
Decision Table
ConditionPhaseStatus
Rule filter matched rule Flagged
AI: IsSafe = true, confidence ≥ 0.6 ai Approved
AI: IsSafe = true, confidence < 0.6 ai Flagged — uncertain
AI: spam or hate, confidence ≥ 0.7 ai Rejected
AI: explicit or harassment, any confidence ai Flagged — human review
AI: parse-error or rate-limited ai Flagged — degraded path
Provider = None (AI disabled) — Approved — rule still runs
Only spam and hate with high confidence auto-Reject. Everything else escalates to Flagged for human review, minimising false-positive auto-removals.
06Persist
Audit Log · Append-only · Never updated
ModerationRecord
One row per decision event: ContentType, ContentId, Status, Phase (rule / ai / human), Provider, Score, Category, Reason, ReviewedById, CreatedOn. Indexed on (ContentType, ContentId) and Status.
Domain · post or reply
SetModerationStatus(status)
Updates moderation_status column on the content row. Both the record and the status update are written in a single db.SaveChangesAsync().
07Outcomes
Approved
  • Content stays visible in feed and search
  • No further automated action
Flagged
  • Content stays visible (surface model)
  • ContentFlaggedIntegrationEvent → admin notified
  • Enters human review queue
Rejected
  • post.SoftDelete() — IsDeleted = true
  • Meilisearch: DeleteAsync(CommunityPosts)
  • Redis: RemoveFromAllFeeds(communityId)
  • ContentFlaggedIntegrationEvent published
Auto-reject fires only when Moderation:AutoRejectOnViolation = true. Default is false — flagged content stays visible until a human confirms.
